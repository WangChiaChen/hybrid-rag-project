## 寫這份設定的三個實測結論

**一、要寫「禁止句」，不要寫「流程描述」。**

| 欄位 | 寫法 | 結果 |
|---|---|---|
| Setting for when unable to answer | 3 條禁止句 | ✅ 乾淨拒答 |
| Generate Response | 5 步驟流程，第 1 步「驗證前提」 | ❌ 仍順著錯誤前提編答案 |
| Scene Check 1 | 科目上下位判斷 | ❌ 只能決定「要不要用這段內容」，擋不住生成階段 |

**二、規則要指名資料裡的欄位，不要只寫抽象描述。**
匯入的 `eap_graph_precise.csv` 有 `累計別`、`單位`、`層級` 這些欄位，
直接叫它去看欄位，比叫它「注意累計值」有效得多。

**三、EAP 的單位換算會算錯，不要讓它換算。** 實測三次全錯：

| EAP 的換算 | 正確值 |
|---|---|
| 10,057 百萬元 →「10.057 億元」 | 100.57 億元 |
| 23,608,252 千元 →「166 億元」 | 236.08 億元 |
| 10,067,506 千元 →「87.6 億元」 | 100.68 億元 |

**四、規則寫在哪一個階段，決定它有沒有效。**

| 階段 | 欄位 | 實測 |
|---|---|---|
| 問題改寫 | Semantic Adjustment | ✅ 有效（D 從失敗轉通過） |
| 拒答判斷 | Setting for when unable to answer | ✅ 有效（B 一直通過） |
| **生成** | **Generate Response** | ❌ **六條規則時 A／C／E 全失敗** |

生成階段的規則要**極少且極短**。六條等於沒有——模型只讀開頭幾句。

**五、規則裡不能出現任何可以拿來運算的數值。**
前一版在 Specific Context A 寫了「基金約占三成多」，
EAP 立刻拿去算：「8,054 × 28% ＝ 推算 2,255 百萬元」，造出財報裡不存在的數字。

### 實測抓到的兩個代表性錯誤

問「為什麼中信 2026Q1 基金手續費收入下滑」（實際是**成長 27.6%**）：

> 2026Q1 財富管理手續費收入為 8,054，較前一季（如 **2025Q3 的 19,860**）明顯下滑

19,860 是 **9M25 三季累計**，8,054 是 3M26 一季累計，拿來相比毫無意義，
卻被用來「證明」一個不存在的下滑。

## 4. Semantic Adjustment（術語對映）

各家對同一指標用詞不一，不對映就會「明明有資料卻查不到」。
實測：問「淨手續費收入」查無，問「手續費淨收益」得到 17,977。

格式盡量貼近平台 Example 的樣式（左為口語／同業用詞，右為文件中的用詞）。
**刻意不放公司簡稱對映**（中信→中信金控之類），那會過度觸發、把不該改的也改掉。

```
若以下詞彙出現在問題中，請替換為右側用詞；若都沒有出現，輸出 "NA"。

淨手續費收入 → 手續費淨收益
淨手收 → 手續費淨收益
手續費收入 → 手續費淨收益
淨利差 → 淨利息收益率
NIM → 淨利息收益率
ROE → 股東權益報酬率
ROA → 資產報酬率
EPS → 每股盈餘
逾放比 → 逾期放款比率
逾期放款比 → 逾期放款比率
覆蓋率 → 備抵呆帳覆蓋率
呆帳覆蓋率 → 備抵呆帳覆蓋率
一銀 → 第一銀行
國泰世華 → 國泰世華銀行
```

> ⚠️ 這欄實測過一次沒生效。若貼上後 D 測試仍失敗，代表平台的替換沒有作用到檢索階段，
> 那就不是提示詞能解決的——改由我們後端在送出問題時附上同義詞（`api._metric_aliases`）。

## 5. Scene Check 1（內容是否足以回答）

這欄只輸出 YES／NO，決定某段內容要不要被採用。
**科目上下位的判斷放這裡沒用**（擋不住生成階段），已移到 Specific Context A。
這裡只保留單純、明確的判斷。

```
你是智能助手，這是是非題，只回答 "YES" 或 "NO"。
判斷[內容]是否包含足以回答[問題項目]的資訊：

1. [內容]與[問題項目]是同一問題、只是換句話說 → YES
2. [內容]能直接解答[問題項目] → YES
3. [問題項目]問某個特定科目時，僅包含其上位／較大科目的[內容]不算
   （例：問「基金手續費」而[內容]只有「財富管理手續費」→ NO）
4. [問題項目]問「原因／為什麼」時，僅有數字而無文字說明的[內容] → NO
5. 只輸出：YES 或 NO
```

## 6. Setting for when unable to answer（查無資料時）

**這欄目前運作正常**，實測能乾淨拒答。維持並補上「不要用推測詞」。

```
當[內容]中找不到足以回答[問題]的依據時：

1. 明確回覆「提供的財報／法說會資料中，未找到／未單獨揭露此項」。
2. 指出缺哪一塊（缺上期比較數字／缺管理層評論／缺同業資料／無此獨立科目）。
3. 嚴禁臆測原因或編造數字，不得使用「可能」「通常」「若」「推測」「應該是」
   等推測性說法來填補缺漏。
4. 不得改用相近但不同的科目數字代答。
```

## 7. Generate Response（生成回答）⭐ 最重要

> ⚠️ **這一欄只能放三條規則，而且要放在最前面。**
> 前一版放了六條絕對規則＋原本的五步驟流程，實測 A／C／E 全部失敗——
> 規則太多等於沒有規則，模型只讀開頭幾句。
> 相對地，`Setting for when unable to answer`（3 條）與 `Semantic Adjustment` 都有效。
> 所以：**只留最致命的三條，其餘規則移到 Specific Context A／B。**

```
# 三條絕對規則（優先於後續所有指示，違反即為錯誤回答）

1. 禁止換算單位、禁止推算數字。
   金額一律照資料原始揭露的單位與數值呈現，不得換算成其他單位，
   不得用佔比或係數推算未揭露的數字。
   跨機構比較時直接說明「單位不同，不宜比較原始數字」，改用比率或每股指標比高下。

2. 只有「累計別」相同的數字能互相比較。
   資料的「累計別」欄位值為「累計」或「單季/當期」。
   兩者不得互比、相減或算增減率，也不得據此推論「下滑」「衰退」「趨緩」。
   都是累計值時，只能比同一季跨年度（3M26 只能比 3M25，不能比 9M25 或 FY25）。

3. 問題若假設了某個趨勢（為什麼下滑／為何衰退），先依規則 2 查證該趨勢是否成立。
   無法合法比較、或資料顯示相反時，第一句就要說「前提與資料不符」並講出實際情形。
   原因只能引用法說會逐字稿中經營團隊的說明，沒有就說「資料中未提供原因說明」，
   不得自行歸因，不得使用「可能」「若」「推測」等說法。

# 以下為回答流程

（接你原本的「原因分析型問題處理流程」5 步驟）
```

## 8. Specific Context A（Industry Keywords）

科目層級定義。**這欄改成禁止句後，C 測試從失敗變成通過。**

> ⚠️ **絕對不要在這裡寫任何百分比或比例。**
> 前一版寫了「基金：財管手續費的組成之一（約占三成多）」，
> EAP 直接拿它當係數：「基金約占 28%，8,054 × 28% ＝ **推算 2,255 百萬元**」，
> 憑空造出一個財報裡不存在的數字。比直接拿母集合數字代答更糟——有推算過程，更像真的。
> 只描述「有哪些組成」，不給任何可以拿來運算的數值。

```
科目定義與層級（回答須據此區分，不得混用）：

- 手續費淨收益／總手續費收入：最上層合計，含財富管理以外的其他手續費。
- 財富管理手續費：中層集合，其組成包含基金、保險代銷、保管及信託、債券等手續費。
- 基金：財富管理手續費的組成項目之一，財報與簡報不單獨揭露其金額。

禁止事項：
- 問「基金手續費」時，不得以「財富管理手續費」的數字作答；
  應回答「財報未單獨揭露基金手續費，僅揭露財富管理手續費合計 X」。
- 問任何子項時，不得以其上位科目的數字代答。
- 禁止用佔比、比例、推估係數去「推算」未揭露的子項金額；
  未揭露就是未揭露，不得以任何方式計算出一個數字。
- 不得將上位科目的變化，說成是子項的變化。
```

## 9. Specific Context B（Industry Know-How）

金控財報的閱讀慣例。原本空白。

```
金控財報的閱讀慣例：

- 各金控申報單位不同（千元／百萬元／億元／十億元），同一份簡報內不同頁也可能不同，
  金額必須連同單位一起呈現。
- 單位換算極易出錯，除非使用者明確要求，否則一律不換算。
  換算時必須寫出算式（例：10,067,506 千元 ÷ 100,000 ＝ 100.68 億元）。
- 法說會簡報的獲利與 EPS 多為「年初至今累計」而非單季，
  名稱可能標示 3M／6M／9M／1H／FY，也可能完全看不出來。
- 新年度第一季的累計數字必然低於前一年第四季的累計數字，那是重新起算，不是衰退。
- 集團（金控）層級與子公司（銀行／人壽／證券／投信）層級的指標不可混用；
  問「金控的 ROE」不得以「銀行的 ROE」作答。
- 「自結數」為未經會計師查核的數字，引用時應一併說明。
- 財報（四大報表）只有數字，不含變化原因；原因只存在於法說會逐字稿。
```

## 與本地系統的連動

改 `Setting for when unable to answer` 的**措辭**時要注意：
本地的查無資料偵測（`api._eap_found_nothing`）靠比對關鍵詞判斷，
措辭一改可能認不出來，Vector RAG 補強與退路按鈕就不會觸發。

目前已涵蓋：查無／查詢不到／查不到／找不到／沒有找到／無法取得／無法提供／
未能查詢／未查詢到／未取得／未找到／尚無／未收錄／無相關資料（資訊／內容），
以及「帶道歉語氣且答案很短」的兜底判斷。

若新增其他說法，請一併更新 `src/api.py` 的 `_EAP_NO_DATA`。

### 這裡設不到的另一道關卡

平台在**檢索之前**還有一道「專案相關性檢查」，被它擋下時回的是英文：

```
Unable to answer question not relevant to this project and its data
```

這句**不受 `Setting for when unable to answer` 控制**——那裡設的是中文，改它不會改到這句。
中文骨架一句都對不上，所以本地一度把它當成「EAP 有回答」，
補強與退路按鈕都不會觸發。現已另立 `_EAP_NO_DATA_EN` 處理（測試 G）。

被這道關卡擋下時，平台其實**根本沒去檢索**，正是 Vector RAG 最該接手的時機：
法說會逐字稿只存在本地，平台那邊本來就沒有。
